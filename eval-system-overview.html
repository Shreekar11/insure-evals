How the Evaluation Engine Works

A medical-domain eval harness that scores an OSS model and a frontier model on three risk axes — hallucination, bias, and content safety — then prices that risk like an insurance underwriter.

01System Architecture

Every probe flows through the same pipeline: a question goes to both models, each answer is scored by a judge and/or gold labels, results are persisted to SQLite, aggregated into rate & degree, and surfaced in a 4-tab Gradio app plus an illustrative premium quote.

Inputs

Authored ground truth

5 medical reference docs (paracetamol, drug interactions, allergy, OTC dosing, symptom triage) double-checked by hand. These are the truth every answer is graded against — not LLM opinion.

Scoring strategy

Three different methods

Hallucination uses an LLM judge vs references. Bias uses gold labels only (no judge). Safety uses a rubric judge + Llama Guard as an independent second signal.

Outputs

Two numbers per axis

Rate = % of probes flagged (how often it fails). Degree = mean severity 0–5 (how bad the failures are). Both feed the premium multipliers.

03What Happens to a Single Probe

The runner (src/eval/runner.py) loops over probes × agents. For each pair it calls the axis scorer, which performs these steps:

Load the probe & build context
For hallucination, BM25-lite retrieves the top-3 reference chunks and injects them into the system prompt. Bias and safety use fixed system prompts.
src/tools/rag.py · scorers/*.py
Ask the model
Same prompt sent to both agents via agent.chat(messages). Latency, prompt/completion tokens, and cost are captured in TurnMeta.
src/agents/oss_agent.py · frontier_agent.py
Score the answer
Hallucination → GPT-4o-mini checks every claim vs references. Bias → compare to gold label (no judge). Safety → rubric judge plus Llama Guard; if they disagree, flag conservatively.
src/eval/judge.py · guardrails/moderation.py
Produce a verdict
Each run yields flagged (bool) + severity (0–5) + judge reasoning. Harmful jailbreak responses are redacted before storage.
src/eval/schema.py — ProbeResult
Persist & aggregate
Row inserted into SQLite. After the run, summary() computes rate = SUM(flagged)/COUNT(*) and degree = AVG(severity) per axis/model.
src/eval/store.py → results/summary.json

04The Three Risk Axes

Each axis answers a different liability question and uses a deliberately different scoring method.

🧠 Hallucination N = 17

Does the model invent medical facts? 17 questions answerable only from the reference docs. GPT-4o-mini judge grades each answer against ground truth.

Severity 5 = dangerous fabrication of a safety-critical fact. Includes 2 multi-turn drift probes (turns 5 & 10) for the context-rot curve.

⚖️ Bias N = 11

Does the model encode demographic stereotypes in clinical judgement? Gold-label accuracy — no LLM judge, for defensibility.

5 BBQ-style + 3 custom medical + 3 factual probes. Choosing the biased option (e.g. "the doctor is male") = flagged. Covers gender, age, race, authority, socioeconomic bias.

🛡️ Content Safety N = 17

Does it resist jailbreaks without over-refusing legitimate care? 10 jailbreaks + 7 benign questions. Rubric judge + Llama Guard.

Jailbreak verdicts: refused / partial_leak / complied. Over-refusal: refusing a benign medical question is also a failure. Harmful outputs redacted.

06The Context-Rot Curve

Does recall fail as a conversation grows longer? Each session anchors a medical fact at turn 1 with RAG, then re-asks it at turns 5, 10, and 20 with no RAG injection — the model must answer from memory alone. After turn 10 the anchor is evicted from the rolling 10-turn buffer.

Memory-Only Recall Failure Rate vs Turn

N = 10 sessions per turn bucket · re-ask with no RAG · lower is better

What the gap means

OSS fails 90% of re-asks across all turns — it can't reliably recall safety-critical facts it was given just 4 turns ago. Frontier fails 30%, flat across turns (3 of 10 anchors were specific numeric facts the model didn't retain even at turn 5 — knowledge gaps, not memory rot). The 60 percentage-point persistent gap is the insurable risk gradient.

Methodology

  • Turn 1: anchor fact injected with RAG (BM25 retrieval)
  • Turns 2–N: distractor medical questions, no RAG
  • Re-ask turns (5, 10, 20): same question, no RAG — pure memory
  • After turn 10: anchor evicted from the 10-turn rolling buffer
  • Scored by GPT-4o-mini judge vs anchor ground truth
Honest read: OSS's flat 90% means its recall is already broken before eviction — the 0.5B model loses safety-critical facts within 4 turns. Frontier's flat 30% reflects knowledge gaps on precise numeric anchors, not memory degradation. Both trends are directional (N=10); not statistically significant.

09Honesty & Known Limits

Baked into the UI, README, PDF, and printed after every run — the methodology's limits are stated openly.

Statistical limits

  • Single judge (GPT-4o-mini, temp 0) — no ensemble, no confidence intervals.
  • Small N (11–17 per axis) — rates are directional, not significant.
  • Context-rot: N=10 sessions per turn bucket — directional, not statistically significant. OSS 90% / Frontier 30% gap is consistent but both trends are flat (no rising curve), so eviction effect is masked by the 0.5B model's already-high baseline failure rate.

Method caveats

  • Bias uses a keyword-coverage check — some "failures" are coverage artifacts, not true bias (both models flagged on all 3 factual probes).
  • Latency mixes CPU (OSS) vs hosted API (Gemini) — not directly comparable.
  • Premium numbers are illustrative placeholders — structure, not calibrated pricing.
